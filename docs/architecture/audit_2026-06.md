# Architecture Audit & Plan — 2026-06

Senior-IC audit of the Playwright Power BI quality suite. Scope of reading: the
helper-functions contract layer, the focus router, one representative enterprise
spec, global setup, and build/test config. No repository-wide rewrite is proposed.

## 1. Intended architectural shift (narrative)

The suite is already split into a **pure logic core** (`refresh-health.ts`,
`source-extraction.ts`, and the routing predicates in `focus.ts`) and an **I/O
edge** (`powerbi-enterprise.ts`, `enterprise-config.ts`, `env-loader.ts`). That
separation is sound and should be preserved, not refactored.

The shift this plan proposes is narrow: **tighten the type and error boundary at
the I/O edge so failures are modeled explicitly, and remove the two remaining
implicit-global seams** — without touching the pure core or the public test
contract (test IDs RH-002/RH-003/VS-NNN, fixture shapes, focus values).

Baseline at audit time: `npm run typecheck` clean; `npm test` (dry-run) 23/23 green.

## 2. Findings

### Strengths (preserve)
- Pure logic is genuinely isolated from I/O: `evaluateRefreshHealth`,
  `analyzeRefreshPatterns`, `scanForDataIntegrityErrors`, `extractFailureInfo`,
  `isInFocus`, `isPqlFocus` take plain data and return plain data
  (`refresh-health.ts`, `focus.ts:131-151`).
- REST layer already uses argument injection: `getAccessToken(credentials,
  endpoints)` and every `list*/find*/get*` takes `accessToken` + `endpoints`
  (`powerbi-enterprise.ts:150-410`). Credentials enter through one seam,
  `readEnterpriseCredentialsFromEnv()`.

### Gaps (ranked by leverage ÷ blast radius)

| # | Finding | Evidence | Risk |
|---|---|---|---|
| G1 | Generic `Error` at every REST failure; no typed domain errors; original cause/stack not attached | `powerbi-enterprise.ts:125-133` | Callers cannot branch on 401/403/404/429; throttling and auth failures are indistinguishable strings |
| G2 | `fetchJson` returns `as T` with zero runtime validation | `powerbi-enterprise.ts:132` | Silent contract drift if Power BI changes a payload |
| G3 | `status` is stringly-typed; closed enum exists; `BAD_STATUSES` duplicated inside spec | `types.ts:8`, `dataset-health.spec.ts:94` | Status logic scattered; typos not caught by compiler |
| G4 | `env-loader.ts` uses a module-level `loaded` singleton and mutates `process.env` | `env-loader.ts:4,39-41` | Hidden init order; not injectable in a unit test without env pollution |
| G5 | `getPowerBiEndpoints` and `readEnterpriseCredentialsFromEnv` read `process.env` via default params | `powerbi-enterprise.ts:61,109-115` | Implicit environment coupling in otherwise-injectable functions |
| G6 | Config/focus path constants resolve `process.cwd()` at import time; spec-level `load*()` calls run file I/O on module load | `enterprise-config.ts:19-24`, `focus.ts:113`, `dataset-health.spec.ts:27-29` | Import-time side effects; cwd-dependent |

None are correctness bugs against the current contract — they are boundary-hardening items.

## 3. Plan (smallest verifiable changes, in order)

Each step is independently shippable, keeps public interfaces, and updates the
narrowest test suite. New behavior is test-first.

1. **Typed REST errors (G1, G2).** ✅ **Shipped 2026-06-05.** Added
   `playwright/helper-functions/errors.ts` with the named `PowerBiError` domain
   error whose closed `kind` discriminant (`auth` | `notFound` | `throttled` |
   `service`) carries `status`, `url`, `body`, and preserves the original
   transport `cause`. `fetchJson` now maps every non-2xx response through
   `httpErrorFromResponse`. Covered by `tests/metadata/errors.spec.ts`
   (ER-001..004), dry-run green (27/27). Deep payload validation (G2) remains
   deferred — see Residual risk.

2. **Status enum (G3).** ✅ **Shipped 2026-06-05.** Added the closed
   `RefreshStatus` union and a single `isBadRefreshStatus()` predicate (one
   source of truth for "bad latest refresh") in `refresh-health.ts`; replaced the
   inline `BAD_STATUSES` set in `dataset-health.spec.ts`. Covered by `RS-001`/
   `RS-002` in `refresh-health.spec.ts`, dry-run green (29/29).

3. **Injectable env loader (G4, G5).** ⏸️ **Deferred — not scheduled.** Analyst
   runs the suite on demand from a workstation; current `process.env` seam is
   acceptable for that workflow. Revisit only if unattended/CI execution is
   introduced.

4. **Import-time I/O containment (G6).** ⏸️ **Deferred — not scheduled.** Same
   rationale as step 3.

5. **(Deferred, already tracked)** Source SQL schema drift (SSD-001..003) and
   unattended CI auth remain as in `work_status.md` Phase 2 — out of scope here.

## 4. Residual risk

- Runtime payload validation (G2) is added only at the fetch boundary; deeper
  field-level schema validation is intentionally deferred to avoid over-modeling
  an external API the suite only reads.
- Steps 3–4 touch module-load ordering for specs; the enterprise lane is
  credential-gated and auto-skips, so regression coverage there is indirect until
  live credentials are available. Mitigation: assert the skip path explicitly.
- No change reduces Fabric CU draw; capacity guidance in `work_status.md` stands.
